# From demo to production

Siddharth Karmarkar, July 2026. Companion to finmo-door1-prototype.html and DESIGN_DECISIONS.md. The design doc defends the choices, this document proves they ship. It maps every interaction in the demo to a concrete build on Finmo's existing stack, AWS hosting, a mix of databases chosen per workload, and Bedrock for AI, with the AI side treated in the most detail because that is where the product thesis lives. Nothing below requires research grade AI or an unproven vendor category. The claim of this document is stronger than feasibility: most of the demo is assembly of patterns that are already boring in production somewhere, and the parts that are not boring are deliberately fenced behind provider interfaces.

## The shape of the whole thing

One platform, organised as services around an event backbone, mirroring the loop from the rebuild report: see, decide, move, learn.

Frontend is a React application served from S3 behind CloudFront, the prototype's render functions map one to one onto components and its single state object splits into server state plus a client store. The API layer is API Gateway in front of services on ECS Fargate for the long lived ones (ledger, payments orchestration, MO runtime) and Lambda for the bursty ones (document processing, webhooks, checks). EventBridge is the backbone: every state transition in onboarding, payments, cards and treasury is an event, which is what makes every later feature an added consumer instead of a schema change. That event spine is the flexibility provision in one sentence.

Data is polyglot on purpose, matching what Finmo already runs. Aurora Postgres holds the ledger and everything that needs transactions and joins: accounts, balances as materialised views over an append only double entry journal, invoices, bills, payouts, cards metadata, treasury holdings. DynamoDB holds high write, key shaped state: onboarding application aggregates, session and wizard progress (the demo's save and resume becomes a row keyed by application id), idempotency keys, rate limit and usage counters for the starter limits, feature flags per tenant. S3 holds every uploaded document, every generated report, immutable and versioned. OpenSearch holds what needs fuzzy search: counterparty names for matching, screening list caches, transaction search. ElastiCache Redis holds FX rates and live quotes. Timestream, or a Postgres partition if the team prefers fewer engines, holds rate history for the risk engine. The rule that resolves every future storage debate: money truth lives in the Postgres journal, everything else is a projection that can be rebuilt from it.

All AI goes through one internal service, the MO runtime, fronting Bedrock. No product service calls Bedrock directly. This is the single most important architectural decision in the document, so it gets its own section before the module walk.

## The MO runtime, one AI front door

The demo shows one MO everywhere: prefilling KYB, checking fields as you type, screening people, scoring applications, matching payins, reading bills, guarding payouts, proposing FX hedges, answering questions. In production that is one service with four internal layers.

The model layer is Bedrock with a routing table, not a single model. Field extraction, categorisation and intent routing run on the smallest model that passes evaluation, Haiku class, at milliseconds and fractions of a cent. Report narratives, matching tie breaks and AskMO conversations run on a mid model, Sonnet class. Nothing in Door 1 needs the largest models. Prompt caching is on for the fixed system prompts, and provisioned throughput is bought only for the two paths with synchronous user facing latency, live checks and AskMO. Every call is logged with model id, prompt hash, token counts and latency, so cost per feature is a dashboard, not an archaeology project.

The tool layer is how MO touches the platform: a registry of typed tools, each one an internal API with a JSON schema, a permission scope and an audit tag. Read tools (get balances, list invoices, get application state) are broad. Write tools are narrow and every one of them terminates in a proposal object, not an action. This is the propose, approve, execute pattern from the demo made structural: the model literally has no tool that moves money, it has tools that create proposals which the product surfaces as the purple cards you see in the prototype, and a human click converts a proposal into a command handled by deterministic services. The two exceptions are standing rules the human already approved, FX rate triggers and auto categorisation, and both write to reversible or advisory state. New agent capability equals new tool registration plus evaluation, no new architecture, which is the AI flexibility provision.

The safety layer is Bedrock Guardrails plus our own: PII redaction before prompts leave the VPC (Bedrock runs within AWS but redaction is still cheap insurance and satisfies data minimisation), injection hardening on every path where model input comes from customer documents or free text (extracted document text is treated as data, wrapped and never concatenated into instructions), output schema validation on every structured call with one retry then deterministic fallback, and a kill switch per tool in AppConfig so any misbehaving capability can be turned off without a deploy.

The evaluation layer is what makes the AI shippable rather than demoable. Every capability has a golden set: a few hundred labelled documents for extraction, labelled payin invoice pairs for matching, labelled transactions for categorisation, a question bank for AskMO. CI runs the sets on every prompt or model change, and the metrics from the rebuild report, proposal acceptance rate, prefill fields kept versus edited, match confirmations, are computed from production events and fed back as new labelled data. The feedback loop the demo gestures at (categorisation learns from corrections) is concretely this: corrections are events, events become labels, labels retune thresholds weekly and few shot exemplars per tenant.

## Module walk

### Onboarding: the KYB and KYC state machines

What the demo shows: a six step business wizard and a four step individual wizard over one machine, save and resume, pause on missing documents, resume on upload.

Production: the application aggregate lives in DynamoDB, the process runs on Step Functions, one state machine definition with the step list injected per product, which is exactly the demo's two step arrays over one engine. Pause and resume is the built in waitForTaskToken pattern: when compliance requests a document the execution parks, the token is stored on the request record, and the upload webhook resumes it. Document upload is S3 presigned URLs from the browser, then an event chain: antivirus scan, Textract for OCR and layout, then MO extraction. Every document state in the demo, uploading, checking, verified, failed with reason, is a status on the document record driven by that chain, and the no silent failure rule is enforced by making the chain emit a terminal event either way, with a dead letter queue and an alarm on anything that emits neither.

The AI: prefill is Textract output plus a Haiku class extraction call with a strict JSON schema per country pack (the same country packs as the demo, stored as configuration), returning value and confidence per field. Fields above threshold prefill with the MO suggested chip, below threshold they stay blank, and the confidence threshold is tuned from the kept versus edited metric. The live checks are a hybrid, deliberately: registration number format and PO box detection are deterministic validators that run client side and again server side, no model involved, because regex problems should not cost tokens or add latency. The website versus description check is AI: a Lambda fetches the site, strips it to text, and a small model call answers one question, does this site plausibly belong to this described business, returning match, mismatch or unclear with a one line reason. Unclear and mismatch are risk signals, never blocks, exactly as in the demo. Feasibility anchor: single field extraction and single question classification are the two most commoditised LLM tasks in existence, and Textract plus Bedrock is the reference implementation AWS themselves document.

### Screening, scoring and the instant approval with limits

What the demo shows: people screened in the background while the form is filled, a KYB report assembled at submit, a numeric score, instant approval with starter limits under threshold, manual review above it, the customer able to read their own report.

Production: sanctions and PEP screening is a vendor integration, ComplyAdvantage or Refinitiv World-Check class, behind a ScreeningProvider interface, called asynchronously the moment a person record saves, results landing as events. An OpenSearch fuzzy prescreen against cached lists gives the instant chip in the UI while the vendor confirms. The scorecard is deterministic and versioned: weighted signals with reason codes, thresholds per country and product stored in DynamoDB configuration, every score reproducible from its inputs. This is a regulatory requirement dressed as a design choice, MAS expects explainable onboarding decisions, so the model does not decide. What the AI does is assemble: a Sonnet class call takes the signal table and writes the analyst facing narrative of the report, the same report the customer sees, with every claim grounded in a signal id so the narrative cannot invent facts. Approval with limits is then pure configuration: the limits object from the demo becomes a policy record referenced by the payments orchestrator, counters for per transfer and per month live in DynamoDB with atomic conditional writes, and the team clears limits by flipping the policy, which is one event. The 1 to 2 day human confirmation is a case in the existing compliance queue, created automatically with the report attached.

Scalability and flexibility: a new market is a new country pack plus scorecard thresholds, data not code. A future ML risk model slots in as one more signal in the scorecard with its own reason code, it does not replace the auditable frame.

### Accounts and collections through J.P. Morgan

What the demo shows: local virtual accounts in eight currencies plus one SWIFT account, issued via JPM, home currency first under starter limits, and PHP, IDR, THB as collect only currencies that settle to USD.

Production: a partner integration service owns the JPM API surface, virtual account issuance, credit notifications by webhook, and daily reconciliation between JPM statements and our journal, with breaks surfacing in an ops queue rather than silently. Issuance is event driven off the approval events, home currency VA on approved_limited, the grid plus SWIFT on approved, exactly the demo's unlock moment. Collect only currencies are a routing table entry: checkout collects local through the acquiring partner, treasury converts, the journal credits USD with the explanatory note carried on the transaction, as the demo does. The provider interface matters here more than anywhere: JPM today, an additional issuer tomorrow, is a second implementation of VirtualAccountProvider, and the eight currency list is configuration.

### Payouts, bills and the payments intelligence

What the demo shows: Bill Pay with MO reading uploaded invoices, a payout flow with source currency, cross currency quotes, duplicate detection requiring acknowledgment, large payment flags, and rail selection with cost against speed.

Production: payouts run as a saga on Step Functions, quote, compliance pre flight, debit, rail submission, settlement confirmation, with idempotency keys on every leg and compensating entries on failure, standard payment orchestration. Rail adapters implement one interface per rail family, the demo's railsFor function becomes a routing service that prices each viable rail and MO's default pick, cheapest that meets the deadline, is a sort, not a model. Bill ingestion reuses the entire onboarding document chain unchanged, S3, Textract, extraction schema for vendor, amount, due date, currency, which is the payoff of building that chain once.

The AI: the pre flight checks are layered exactly as the demo implies. Duplicate detection is deterministic first, same counterparty and amount within a window is a database query, then an embedding similarity pass over counterparty names through Bedrock embeddings stored in OpenSearch catches Acme Pte Ltd versus ACME PTE, and the model only writes the human explanation on the warning card. Anomaly flags start as statistics, size against the customer's own trailing distribution, new beneficiary, unusual hour, each a scored signal with the same reason code discipline as onboarding, and a learned model can replace the statistics later behind the same signal interface. The design rule across all payments intelligence: detection is deterministic or statistical and cheap, language is where the LLM spends, because a wrong block costs customer trust and a wrong explanation costs nothing but a reword.

### Collections intelligence: invoices, matching and links

What the demo shows: invoice module, MO matching an incoming payin to an open invoice with one click confirmation, nudges drafted for approval, payment links with the collect only settlement note.

Production: matching is candidate generation then ranking then proposal. Candidates come from SQL, open invoices within an amount and date window. Ranking is embedding similarity on names plus amount and reference heuristics. Only ambiguous top pairs go to a model call for a tie break with stated confidence, and everything lands as a proposal event the UI renders as the match card. Confirmations and rejections are labels, thresholds retune from them, and auto match above a very high confidence can be enabled per tenant later as a standing rule, the same consent pattern as FX triggers. Nudges are a Sonnet draft from invoice facts and tenant tone settings, queued for approval, sent through the existing notification service. Payment links are the checkout page with a token, a hosted page on CloudFront, a link record in Postgres, and the acquirer webhook driving the same credit path as VAs. No AI is needed to make links work, which is worth saying in a document about AI: part of proving feasibility is knowing where not to spend a model.

### Cards and expense management

What the demo shows: instant virtual cards with limits and category locks, plastic on request, spend auto categorised, receipts chased by MO.

Production: an issuer processor, Marqeta or Lithic class, behind a CardProvider interface, with authorisation webhooks flowing into the journal in real time. Spend controls, the demo's monthly limit and category lock, evaluate in the authorisation callback against policy records, deterministic and fast because the network gives tens of milliseconds. Categorisation is MCC and merchant rules first, covering most volume for free, then embeddings against the tenant's own corrected history, then a small model call for the long tail, with corrections feeding back weekly as few shot exemplars per tenant. Receipt chasing is an MO standing rule: a missing receipt event schedules a drafted email, tenant configurable, and an inbound receipt image runs through the same document chain and attaches by amount and date match. Cards stay gated to full approval as in the demo, which in production is just the card policy referencing the same limits object as payments.

### Treasury on the Synthesys Network

What the demo shows: a treasury account open from day one, the real fund catalogue with real minimums and settlement windows, movement only between business account and treasury, daily visible yield, and a disclosure gate for higher risk funds.

Production: Finmo is sub distributor, Synthesys holds the principal and client money accounts with the fund managers, and the load bearing consequence is that Finmo's journal is the source of truth for customer positions, an omnibus holding on the network mapped to per customer rows in our ledger. That is the same safeguarding discipline the e money business already runs, extended to fund units, and it is why the only allowed movement is business account to treasury and back, enforced in the transfer service, not the UI. The order service batches subscriptions and redemptions against the 14:00 SGT cutoff with an EventBridge schedule, tracks T+0 through monthly settlement states per fund from catalogue configuration, and ingests NAV and yield feeds daily to post accruals as journal entries, which is what makes yesterday's interest a real ledger line rather than a display trick. The suitability gate stores each accepted disclosure with document version, timestamp and account, immutable in S3, because when a higher risk fund has a bad quarter the evidence of informed consent is the product. The fund catalogue itself is configuration synced from the network, adding a fund touches no code. AI here is deliberately thin: AskMO answers yield and holdings questions from the ledger, and the idle cash suggestion on the home page is a threshold rule with a model written sentence, because investment advice is a regulated act and a suggestion engine that stays on the right side of that line is a product decision the AI must inherit, not blur.

### FX and the VaR engine

What the demo shows: exposures assembled from the customer's own balances and receivables, parametric and Monte Carlo VaR with toggles for confidence, horizon and basis, component bars, scenario chips, and MO proposals ending in a conversion or a target rate trigger.

Production: the demo's pure engine ports almost line for line into a risk service, the one substantive upgrade is the covariance input. Rates stream from the market data provider into Redis for quoting and into Timestream for history, a nightly job computes EWMA volatilities and correlations per tenant relevant currency set, and the engine takes a covariance provider interface so the demo's representative numbers and production's estimated ones are two implementations of the same signature. VaR computes on demand for the FX screen, cheap enough at Door 1 portfolio sizes to be synchronous, and nightly in batch for the home tile and for backtesting, where realised P&L is compared against the forecast band and the hit rate is published to the customer, the trust move the rebuild report demanded of forecasting in general. Triggers are standing orders in Postgres evaluated against the rate stream by a small always on consumer, with balance pre checks, expiry, notification on fire or fail, and the same loud failure rule as everywhere else. The cash tools version for Door 2 is this same service with position sources widened to connected bank accounts and forecast flows, which is precisely why the engine was built UI free. The AI: exposure assembly and mathematics are deterministic, MO's role is translation and proposal, the plain words headline, the hedge suggestion with its quantified VaR reduction, both generated from engine output with the numbers passed through, never recomputed by the model.

### AskMO

What the demo shows: one entry point, curated prompts that always work, grounded answers about balances, limits, bills, risk and treasury, graceful boundaries.

Production: AskMO is the conversational face of the MO runtime, intent routing on a small model, tool calls against the read registry, answers grounded in tool results with the rule that any number in a response must arrive from a tool, not from generation. The curated prompts are registered against health checks, a prompt whose backing service is degraded is swapped out of the suggestion row before a customer can click it, which turns the report's sharpest finding, the marquee prompt that failed, into a structural impossibility. Conversation state lives in DynamoDB with a short TTL, long term memory is only the correction and preference data that flows through the evaluation layer, and everything the runtime does lands in the same audit log as the rest of the platform, per tool, per proposal, per approval, the log the trust metric is computed from.

## Cross cutting: what makes it scale and what keeps it flexible

Multi tenancy and isolation: tenant id on every row and every event, IAM boundaries per service, and the doors of the product, business, remit, and the future treasury led Door 2, are view configuration over the same services, which the demo already demonstrates by running two products off one state machine and one ledger.

Scale path: the ledger partitions by tenant and posts through an append only journal, so read load goes to projections and caches while the write path stays narrow. Everything AI is asynchronous except live checks and chat, so Bedrock throughput is a queue depth question, not an availability question. Step Functions and Lambda absorb onboarding and document spikes without capacity planning. The expensive well known problems, card authorisation latency and rate stream fan out, sit in isolated services that can be scaled or rewritten without touching the rest.

Flexibility provisions, concretely: provider interfaces for screening, virtual accounts, card issuing, acquiring, fund network and market data, so every partner is swappable and every second partner is an implementation, not a redesign. Country packs, scorecards, limits, fund catalogue and rail tables as configuration. The event backbone for new consumers, a lending product, a Door 2 forecaster, an accounting sync, each subscribes to events that already exist. And the MO tool registry, where every new AI capability is a tool plus an evaluation set, behind the same proposal discipline.

Compliance posture, briefly: data residency per region using regional AWS deployments under the existing licence map, PDPA and MAS TRM alignment through the audit log, immutable documents and reproducible decisions, model risk management satisfied by the deterministic scorecard frame with AI in assembly and language roles, and human accountability preserved by the proposal pattern everywhere money moves.

## Sequencing and the honest risks

A credible order of build: the document chain and MO runtime first, because onboarding, bills and receipts all ride on it. Then the scored onboarding with limits, then JPM accounts and the payout saga, then matching and categorisation which are the highest visible AI value per unit effort, then cards, then treasury, then the risk service. Each stage ships customer visible value alone.

The honest risks are not AI. They are partner timelines, JPM onboarding, issuer processor contracts, Synthesys integration depth, and the operational load of reconciliation and compliance queues. The AI components are the low risk part of this document, every one of them is a pattern with public production precedent on Bedrock, and every one of them degrades gracefully to a manual path because the manual path is what the proposal pattern falls back to by construction. That is the feasibility argument in one line: the demo's AI is ambitious in product terms and conservative in engineering terms, which is exactly the right place for a licensed financial institution to put its ambition.
